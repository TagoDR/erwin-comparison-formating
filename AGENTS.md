# Project: Erwin Compare Formatter

## 1. Overview
The **Erwin Compare Formatter** is a specialized utility designed to ingest HTML difference reports generated by Erwin Data Modeler. It transforms the raw, often difficult-to-read comparison data into a highly visual, searchable, and filterable interface optimized for human analysts.

The final output is a single-file HTML application (built via Vite) that can be easily shared and opened in any modern web browser.

## 2. Technical Stack
- **Framework:** Vite with TypeScript
- **State management** nanostore for lit
- **Output Type:** Single-file HTML (using `vite-plugin-singlefile`)
- **UI Components:** [Web Awesome](https://webawesome.com/)
- **Target Language:** Portuguese (pt_BR) for user interface instructions.
- **Linting** Biome
- **Testing** no unit testing only storybook for visuals

## 3. Core Functionality

### 3.1. Data Ingestion
- Users provide an HTML file generated by Erwin's "Complete Compare" tool.
- The system parses the source table which typically consists of three main columns:
    1. **Type/Object:** The object name and its properties (nesting indicated by space-based indentation).
    2. **Left Model:** The "Working" or "Current" model state.
    3. **Right Model:** The "Reference" or "Baseline" model state.

### 3.2. Data Processing & Enhancement
The parser must extract and "hoist" information to create a enriched data model:
- **Nesting Logic:** Interpret leading spaces in the Type column to determine parent-child relationships (e.g., Table -> Column -> Index).
- **Change Detection (Hoisting):**
    - **I (Inclusion):** New objects (exists in Left, not in Right).
    - **A (Alteration):** Changed objects (exists in both, but properties differ).
    - **E (Exclusion):** Deleted objects (exists in Right, not in Left).
    - *Note:* Change status must be hoisted from property differences up to the Data Object Identifier row.
- **View Flagging:** Detect "Logical Only" and "Physical Only" properties and hoist this information to the identifier row for quick visibility.

### 3.3. Column Mapping
The formatted output table will include:
| Column | Description |
| :--- | :--- |
| **Type** | The indented hierarchy of objects and properties. |
| **Prop** | Identifier grouping (replicated across all property rows of the object). |
| **Change** | Status indicator: **I** (Inclusion), **A** (Alteration), or **E** (Exclusion). |
| **View** | Flags for "Logical Only" or "Physical Only". |
| **Left Model** | Data from the working model. |
| **Right Model** | Data from the reference model. |

## 4. UI/UX Requirements

### 4.1. Layout
- **Sticky Header:** A slim top bar containing:
    - File input for HTML ingestion (Instructions in **pt_BR**: e.g., "Selecione o arquivo HTML do Erwin").
    - Filter controls (Change Type, Object Type).
- **Main Display:** A scrollable area containing the enhanced data table.

### 4.2. Visual Coding (Color Scheme)
Objects are color-coded based on their type and status to facilitate rapid scanning:

#### Tables
- **New Table:** Dark Green
- **Changed Table:** Dark Purple
- **Deleted Table:** Dark Red

#### Columns
- **New Column:** Green
- **Changed Column:** Purple
- **Deleted Column:** Red

#### Other Objects (Index, Tablespace, View, etc.)
- **Color:** Orange
- **Visual Cues:** Decreasing color intensity as nesting depth increases.

### 4.3. Filtering Capabilities
- **By Change Type:** Show only inclusions, exclusions, or alterations.
- **By Object Category:** Filter between "Table" and "Others" (Indexes, Views, etc.).

## 5. File Structure & Components
- `src/parser/`: Logic for processing the Erwin HTML structure.
- `src/components/`: Web Awesome based UI components.
- `src/store/`: State management for filtered data.
- `llms.txt`: Reference for Web Awesome implementation details.
